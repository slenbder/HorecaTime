# HorecaTime — Telegram-бот учёта рабочего времени

## Суть проекта
Перенос Telegram-бота с Google Apps Script на Python (aiogram 3.x).
Apps Script версия — рабочий референс в продакшне. Файлы скриптов доступны в ассетах проекта.

---

## Стек
- Python 3.11+, aiogram 3.13.1
- gspread 6.1.2, oauth2client 4.1.3
- APScheduler 3.10.4
- aiosqlite 0.20.0
- SQLite — только FSM + кеш ролей, НЕ основная БД
- Google Sheets — единственный источник данных (single source of truth)

## Логирование — ОБЯЗАТЕЛЬНЫЕ ПРАВИЛА

Три лог-файла, настроены в `app/logging_config.py`:
- `logs/app.log` — общий поток: старт, действия пользователей, успешные операции
- `logs/errors.log` — только ошибки и исключения
- `logs/googleapi.log` — все обращения к Google Sheets (запросы, ответы, реконнекты)

**Всегда логировать:**
- Старт и остановку бота
- Каждый вызов Google Sheets API (в googleapi.log)
- Реконнекты (`_reconnect()`)
- Approve/reject пользователя
- Добавление пользователя в месячный лист
- Критические ошибки → лог + Telegram-алерт разработчику

**Никогда не использовать** `print()` — только `logging`.

## Google Sheets — структура

**Техлист** (колонки):
- A(1): Telegram ID
- B(2): @Ник
- C(3): ФИО от пользователя
- D(4): Отдел (Зал/Бар/Кухня)
- E(5): Позиция
- F(6): Дата регистрации (DD.MM.YY HH:MM, Moscow time)
- G(7): «ДА» — утверждён в графике

**Месячный лист** (формат названия: «Март 2026», т.е. `{MONTH_NAMES_RU[month]} {year}`; название «Часы» **не используется** в новой версии бота):
- A: ФИО, B: Telegram ID, C: позиция
- Строка 3: даты месяца
- Данные: D5:R60 и T5:AI60
- Ключевые ячейки: C2 (месяц), T2 (год)

Итоговые колонки (формат H/AH, формулы вставляются ботом при добавлении сотрудника):
- S(19): H/AH первая половина (дни 1-15)
- AJ(36): H/AH вторая половина (дни 16-конец)
- AK(37): H/AH за месяц

Служебные колонки для Раннера (бот пишет напрямую, числовой формат):
- AL(38): итого выходных H за месяц (формула =AM+AN, вставляется при добавлении)
- AM(39): выходные H за первую половину (дни 1-15), накапливается при write_shift
- AN(40): выходные H за вторую половину (дни 16-конец), накапливается при write_shift

Текстовый формат (TEXT) задаётся для диапазона D:AK; AL/AM/AN остаются числовыми.

## Бизнес-логика — ключевые правила

**Расчёт часов:**
- `H = endTime - startTime`, округление до 0.5 (`roundToHalf`)
- Пересечение полуночи обрабатывается: `H = 24 - start + end`
- Выходные: пятница, суббота, воскресенье — ставка выходного дня

**Форматы ввода даты:** `1.1` / `01.01` / `1.01.25` / `01.01.25`
**Форматы ввода времени:** `1000-1800` / `10-18` / `10.00-18.00`
**Спецзначения:** `X` = рабочий день (без часов), `О/o/0` = нулевая смена

**Добавление в месячный лист:**
1. Найти секцию по позиции (`POSITION_TO_SECTION`)
2. Fallback: конец блока отдела (`DEPARTMENT_TO_HEADER`)
3. Fallback: конец листа
4. Защита от дублей — проверка по Telegram ID в колонке B

## Механика H/AH по позициям

Формат ячейки в месячном листе: H/AH (например 8/3). Если AH=0 → пишем H/0.
Нулевая смена → пустая ячейка. Спецзначение X — не используется.

| Позиция                | H  | AH                                          | Повышенная ставка (пт/сб/вс) | Ввод нескольких смен |
|------------------------|----|---------------------------------------------|------------------------------|----------------------|
| Кухня (все цеха)       | ✅ | ❌                                          | ❌                           | Одним сообщением     |
| Грузчик/Закупщик       | ✅ | ❌                                          | ❌                           | Одним сообщением     |
| МОП (Клининг/Котломой) | ✅ | ❌                                          | ❌                           | Одним сообщением     |
| Хостесс                | ✅ | ❌                                          | ❌                           | Одним сообщением     |
| Менеджер               | ✅ | ❌                                          | ❌ (уточнить позже)          | Пошагово             |
| Официант               | ✅ | ✅ карты/чеки (0.5ч/фото, апрув admin_hall) | ❌                           | Пошагово             |
| Раннер                 | ✅ | ✅ доп.часы через бота (всегда базовая)     | ✅ только H                  | Пошагово             |
| Бармен/Барбэк          | ✅ | ✅ тусовочные (проверка нахлёста)           | ❌                           | Пошагово             |

### Детали по позициям

**Раннер:**
- H = часы смены, округление до 0.5
- Повышенная ставка на H в пт/сб/вс (определяется по дате на Python)
- AH = доп.часы, вводит пользователь через бота после основной смены
- После ввода AH бот запрашивает комментарий, который включается в уведомление админу
- AH всегда оплачиваются по базовой ставке, даже в выходные
- Запись: H/AH (например 8/5). В уведомлении админам: H отдельно, AH отдельно с комментарием

**Официант:**
- H = часы смены, одна базовая ставка (без повышенной в выходные)
- AH = доп.часы за карты лояльности и чеки (каждое фото = 0.5ч AH)
- Механика:
  1. Официант вносит смену + прикрепляет N фото одним сообщением
  2. Бот отправляет admin_hall медиагруппу (фото) + отдельное сообщение:
     👤 ФИО — Официант
     📅 дата
     ⏱ время → H = X ч
     📎 Приложено фото: N
     ✅ Сколько фото засчитать как доп. часы?
     [0] [1] [2] ... [N]
  3. callback_data кнопки: `approve_ah:{telegram_id}:{date}:{value}`
  4. Админ нажимает кнопку → бот пишет AH в таблицу
  5. Сообщение редактируется через editMessageText: кнопки исчезают, в конце добавляется строка "✅ Одобрено X фото из N → AH = Y ч"
  6. Официанту уведомление: "📋 Смена {date} обработана\nH = X ч | AH = Y ч (X фото из N)"
- Inline keyboard привязана к отдельному сообщению (не к медиагруппе — Telegram не поддерживает)
- Каждый отчёт независим, админ нажимает кнопки в любом порядке
- Запись: H/AH

**Бармен/Барбэк:**
- H = обычные часы смены (базовая ставка)
- AH = тусовочные часы (повышенная/тусовочная ставка)
- День недели не влияет на ставки
- Проверка нахлёста: H и AH не могут пересекаться по времени
- При нахлёсте → алерт пользователю с просьбой исправить диапазоны
- Запись: H/AH (например 3/9). Если тусовки не было → H/0

**Кухня / Хостесс:**
- Только H, одна ставка, без повышенных и без AH
- Несколько смен можно внести одним сообщением (несколько строк)

**Менеджер:**
- Пока как Кухня/Хостесс (только H), детали уточнятся позже

## Важные договорённости

- **Не использовать** `print()` нигде в коде
- **Не менять** структуру таблиц Google Sheets — это продакшн
- **Circular import**: ID ролей импортировать из `config.py`, не из `auth.py`
- **Apps Script файлы** в ассетах проекта — референс логики, не трогать
- **Git workflow**: ветки от `main`, PR без автомерджа, коммит после каждой задачи
- Название месячных листов: `{MONTH_NAMES_RU[month]} {year}` (например «Март 2025»)
- **Уведомления о сменах** всегда отправляются: admin своего отдела + все superadmin
  (Зал: `ADMIN_HALL_IDS + SUPERADMIN_IDS`, Бар: `ADMIN_BAR_IDS + SUPERADMIN_IDS`, Кухня: `ADMIN_KITCHEN_IDS + SUPERADMIN_IDS`; дубли исключать через `set`)
- **Увольнение**: ячейка A уволенного сотрудника в текущем месячном листе красится в `#FFCCCC`
- Красные строки **НЕ переносятся** в новый месячный лист (реализуется в `monthly_switch.py`)
- Из Техлиста строка **удаляется**, из месячного листа — **нет** (история часов сохраняется)
- **approve_ah_callback** должен быть зарегистрирован ДО generic `approve_`/`reject_` хендлеров в `auth.py`
- **Формат ячеек месячного листа**: колонки D:AN должны иметь формат "Обычный текст" (настраивается вручную в таблице) — иначе числа интерпретируются как даты
- **Проверка прав superadmin/developer** — через `SUPERADMIN_IDS`/`DEVELOPER_ID` из `config.py`, не через SQLite (суперадмины не регистрируются в таблице `users`)
- **Администраторы отделов** регистрируются как обычные сотрудники, повышаются суперадмином через `/promote`. После повышения сотрудник вводит Gmail — суперадмин вручную добавляет его в редакторы таблицы.
- **При `/demote`** суперадмину напоминают удалить email из редакторов таблицы.
- **`/dismiss` для действующего admin** показывает развилку: "Только понизить / Уволить / Отмена"
- **Ставки персональные** (таблица `user_rates`); `rates` остаётся шаблоном для новых сотрудников — не используется для расчёта зарплаты напрямую
- **При `switch_month()`** делаются снимки ставок: `rates_history` (шаблонные) и `user_rates_history` (персональные) — оба до копирования листа
- **Шеф/Су-шеф** вводят должность вручную (`custom_title`) → пишется в колонку C месячного листа
- **Грузчик/Закупщик** — позиции отдела «Доп.» (Кухня); в Техлисте хранится фактическая должность («Грузчик» или «Закупщик»), секция листа — «Дополнительные сотрудники»
- **МОП** (Клининг, Котломой) — отдельный отдел; подчиняется `admin_hall` (уведомления о сменах и заявках → `ADMIN_HALL_IDS + SUPERADMIN_IDS`)
- **callback_data** кнопок не должен превышать 64 байта (лимит Telegram); данные заявки администратора хранятся в `_pending_admins`, а не в callback_data
- **Email администратора** при регистрации — только `@gmail.com` (валидация через `_is_valid_gmail()`)
