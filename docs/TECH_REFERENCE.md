## Терминология (критично важно!)

**Отдел (department)** — подразделение в ресторане:
- Зал
- Бар
- Кухня
- МОП

**Позиция (position)** — роль сотрудника в отделе:
- Официант, Раннер, Хостесс, Менеджер (Зал)
- Бармен, Барбэк (Бар)
- Руководящий состав, Горячий цех, Холодный цех и т.д. (Кухня)
- Клининг, Котломой (МОП)

**Должность (custom_position)** — персональное название (ранее называлось `custom_position`):
- Используется только для Руководящий состав (например "Шеф ЗЦ", "Бренд-шеф")
- И для Грузчик/Закупщик (например "Грузчик Петров")
- Хранится в колонке **H Техлиста** (отдельно от позиции в E)

## Фантомный сотрудник (Наполняемость чека)

### Общие сведения

**Назначение:** Накопление одобренных наполняемостей чеков для последующего ручного распределения менеджером.

**TG_ID:** 1984002026 (хардкод в `config.py`)  
**Имя:** "Наполняемость чека"  
**Ставка:** 1500 р/час (хардкод в `config.py`)

### Регистрация

- **НЕ проходит** через approve flow
- **НЕ в Техлисте**
- **НЕ в SQLite** `users`
- Создаётся **вручную** администратором в месячном листе
- Колонка B месячного листа должна иметь формат **"Обычный текст"** (не Автоматический). Без этого switch_month не найдёт фантома.

### Позиционирование

- **Отдел:** Зал
- **Позиция:** Официант (для попадания в правильный блок)
- **Место в листе:** Первая строка в секции "Официанты"

### Запись данных

**Формат ячейки:** Только число (например "5")

**Алгоритм:**
1. Админ одобряет N чеков официанта
2. Читается текущее значение ячейки дня: `current = 2`
3. Суммируется: `new = current + N = 2 + 3 = 5`
4. Записывается: `"5"` (RAW)

**Важно:** Суммирование в пределах **одного дня**. Несколько официантов за один день → накопление.

### Формулы

**Колонка S (19)** — первая половина месяца:
```
=SUM(D{row}:R{row})
```

**Колонка AJ (36)** — вторая половина:
```
=SUM(T{row}:AI{row})
```

**Колонка AK (37)** — весь месяц:
```
=S{row}+AJ{row}
```

**Отличие от обычных сотрудников:** Формулы простые SUM (без разделения H/AH).

### Switch Month

**Перенос:**
- Фантом **переносится** в новый месяц
- H/AH **обнулены**
- Позиция **сохраняется** (первая строка в секции Официантов)
- Формулы **вставляются** заново

**Функция:** `_transfer_phantom_to_new_month()`

**Если не найден:** Warning в лог, switch_month продолжается (не падает).

### Отображение

**В `/hours_*` для Официанта:**
```
💳 Общий пул чеков: 47 шт (70500 р)
```

**Показывается:**
- Даже если пул = 0 (формат: "0 шт (0 р)")
- Только для позиции "Официант"
- Другие позиции пул не видят

### Распределение

**Текущая реализация:** Нет автоматики.

**Планируемое:**
- Ручное распределение менеджером в конце месяца
- UI для управления распределением (будущая задача)

## Структура Google Sheets

### Техлист (колонки)

| Колонка | Название | Описание |
|---------|----------|----------|
| A (1) | Telegram ID | Уникальный идентификатор пользователя |
| B (2) | @Ник | Username в Telegram |
| C (3) | ФИО | Полное имя от пользователя |
| D (4) | Отдел | Зал/Бар/Кухня/МОП |
| E (5) | Позиция | Базовая позиция (всегда каноническая: "Руководящий состав", "Официант" и т.д.) |
| F (6) | Дата регистрации | DD.MM.YY HH:MM (Moscow time) |
| G (7) | Статус | "ДА" — утверждён в графике |
| H (8) | Должность | `custom_position` для РС/Грузчик/Закупщик (пусто для остальных) |

### Месячный лист

**Формат названия:** `{MONTH_NAMES_RU[month]} {year}` (например "Март 2026")

**Структура:**
- **A:** ФИО
- **B:** Telegram ID
- **C:** Позиция (или custom_position для Руководящий состав)
- **Строка 3:** Даты месяца
- **Данные смен:** D5:R60 (дни 1-15), T5:AI60 (дни 16-конец)
- **Ключевые ячейки:** C2 (месяц), T2 (год)

**Итоговые колонки (формат H/AH):**
- **S (19):** H/AH первая половина месяца (дни 1-15)
- **AJ (36):** H/AH вторая половина месяца (дни 16-конец)
- **AK (37):** H/AH за весь месяц

**Служебные колонки для Раннера (числовой формат):**
- **AL (38):** Итого выходных H за месяц (формула =AM+AN, вставляется при добавлении)
- **AM (39):** Выходные H за первую половину (дни 1-15), накапливается при write_shift
- **AN (40):** Выходные H за вторую половину (дни 16-конец), накапливается при write_shift

**write_shift() возвращает:** `old_value` (str) — предыдущее содержимое ячейки до записи, или `''` если ячейка была пустой. Используется в userhours.py для определения факта перезаписи и уведомления администраторов.

**Формат ячеек:**
- **B (TG_ID)** = "Обычный текст" (TEXT) обязателен — аналогично D:AK; иначе научная нотация ломает поиск по TG_ID в switch_month
- **D:AK** = "Обычный текст" (TEXT) — настраивается вручную в таблице
- **AL/AM/AN** = числовой формат для корректной работы формул

**Защита от инъекций:**
- **value_input_option="RAW"** для всех пользовательских данных (защита от formula injection)
- **value_input_option="USER_ENTERED"** только для hardcoded формул (`=SUMPRODUCT`, `=S+AJ` и др.)
- Защита от: `=HYPERLINK()`, `=IMPORTXML()`, `=IMPORTDATA()`

## SQLite — схемы таблиц

### Таблица `users`

```sql
CREATE TABLE users (
    telegram_id INTEGER PRIMARY KEY,
    full_name   TEXT NOT NULL,
    role        TEXT NOT NULL,  -- user/admin_hall/admin_bar/admin_kitchen/superadmin/developer
    department  TEXT,           -- Зал/Бар/Кухня/МОП
    position    TEXT,           -- Официант/Раннер/Бармен/... (заполняется при approve)
    created_at  TEXT NOT NULL
);
```

**Важно:** Поле `hourly_rate` удалено — ставки теперь в `user_rates`.

### Таблица `fsm_storage`

```sql
CREATE TABLE fsm_storage (
    chat_id INTEGER NOT NULL,
    user_id INTEGER NOT NULL,
    bot_id  INTEGER NOT NULL,
    state   TEXT,
    data    TEXT NOT NULL DEFAULT '{}',
    PRIMARY KEY (chat_id, user_id, bot_id)
);
```

### Таблица `user_rates` (персональные ставки)

```sql
CREATE TABLE user_rates (
    telegram_id  INTEGER PRIMARY KEY,
    base_rate    REAL NOT NULL,
    extra_rate   REAL,           -- NULL для позиций без повышенной ставки
    updated_at   TEXT NOT NULL,
    FOREIGN KEY (telegram_id) REFERENCES users(telegram_id) ON DELETE CASCADE
);
```

**Критично важно:**
- `user_rates` = **единственный источник** ставок сотрудников
- При апруве нового сотрудника ставка **НЕ устанавливается** автоматически
- Администратор устанавливает ставку через `/set_rate` вручную
- Новый сотрудник **не имеет** записи в `user_rates` до первого `/set_rate`

**Использование:**
- У каждого сотрудника своя персональная ставка
- Редактируется через `/set_rate` (admin)
- Используется для расчёта зарплаты в `/hours_*`

**`get_users_rates_by_department` использует LEFT JOIN:**
- Возвращает **всех** сотрудников отдела, включая тех, у кого нет записи в `user_rates`
- Для новых сотрудников `base_rate` и `extra_rate` будут `None`
- Это позволяет `/set_rate` видеть нового сотрудника и назначить ему ставку с нуля
- До этого фикса был INNER JOIN — новые сотрудники выпадали из списка (замкнутый круг)

### Таблица `user_rates_future` (запланированные изменения ставок)

```sql
CREATE TABLE IF NOT EXISTS user_rates_future (
    telegram_id    INTEGER PRIMARY KEY REFERENCES users(telegram_id) ON DELETE CASCADE,
    base_rate      REAL NOT NULL,
    extra_rate     REAL,
    effective_date TEXT NOT NULL,  -- формат YYYY-MM-DD
    created_at     TEXT NOT NULL
);
```

**Использование:**
- Позволяет запланировать изменение ставки "со следующего месяца"
- Устанавливается через `/set_rate` → выбор "Со следующего месяца"
- Применяется автоматически при `switch_month()` через `apply_future_rates()`
- После применения запись удаляется из `user_rates_future`

**Пример:**
```sql
-- Сегодня апрель 2026, ставка Иванова 250р
-- Планируем повышение до 300р с 1 мая
INSERT INTO user_rates_future VALUES (12345, 300.0, NULL, '2026-05-01', '2026-04-15 10:00:00');

-- При switch_month() (1 мая):
UPDATE user_rates SET base_rate = 300.0 WHERE telegram_id = 12345;
DELETE FROM user_rates_future WHERE telegram_id = 12345;
```

### Таблица `user_rates_history` (снимки при переключении месяца)

```sql
CREATE TABLE user_rates_history (
    telegram_id  INTEGER NOT NULL,
    base_rate    REAL NOT NULL,
    extra_rate   REAL,
    month        INTEGER NOT NULL,
    year         INTEGER NOT NULL,
    PRIMARY KEY (telegram_id, month, year)
);
```

**Использование:**
- Снимок делается при `switch_month()`
- Используется в `/hours_last` для расчёта зарплаты за прошлый месяц
- Защита от изменений ставок "задним числом"

### Таблица `rates_history` (legacy, для старых данных)

```sql
CREATE TABLE rates_history (
    position     TEXT NOT NULL,
    base_rate    REAL NOT NULL,
    extra_rate   REAL,
    month        INTEGER NOT NULL,
    year         INTEGER NOT NULL,
    PRIMARY KEY (position, month, year)
);
```

**Использование:**
- Хранит старые данные до миграции на `user_rates`
- Fallback в `/hours_last` если нет данных в `user_rates_history`

## Логика расчёта зарплаты

### Официант, Менеджер, Хостесс, Кухня (кроме Раннера)

```python
user_rate = get_user_rate(telegram_id)
salary = total_hours * user_rate.base_rate + ah_hours * user_rate.base_rate
```

### Раннер

```python
user_rate = get_user_rate(telegram_id)
regular_hours = total_hours - weekend_h_hours
salary = regular_hours * user_rate.base_rate + weekend_h_hours * user_rate.extra_rate
```

**Важно:** Выходные часы (пт/сб/вс) считаются по `extra_rate`, будние по `base_rate`.

### Бармен / Барбэк

```python
salary = h * base_rate + ah * (extra_rate if extra_rate else base_rate)
```

**Важно:** Все обычные смены (H) по `base_rate`, тусовочные часы (AH) по `extra_rate`. Никакого порога в 60 часов нет.

## Формулы месячного листа

При добавлении нового сотрудника в месячный лист автоматически вставляются формулы:

**Ячейка S (колонка 19) — первая половина месяца:**
```
=D{row}&IF(E{row}<>"","/"&E{row},"")&IF(F{row}<>"","/"&F{row},"")&...&IF(R{row}<>"","/"&R{row},"")
```

**Ячейка AJ (колонка 36) — вторая половина месяца:**
```
=T{row}&IF(U{row}<>"","/"&U{row},"")&IF(V{row}<>"","/"&V{row},"")&...&IF(AI{row}<>"","/"&AI{row},"")
```

**Ячейка AK (колонка 37) — весь месяц:**
```
=S{row}&IF(AJ{row}<>"","/"&AJ{row},"")
```

**Ячейка AL (колонка 38) — итого выходных H для Раннера:**
```
=AM{row}+AN{row}
```

## FSM State Machine — список состояний

### AuthStates (регистрация и управление)

```python
# Регистрация
choosing_department     # Выбор отдела (Зал/Бар/Кухня/МОП)
choosing_position       # Выбор позиции в отделе
entering_fio            # Ввод ФИО текстом

# Кухня
waiting_kitchen_title   # Ввод должности для Руководящий состав
waiting_dop_position    # Выбор Грузчик/Закупщик

# Dismiss (3 шага)
waiting_dismiss_dept_type   # Выбор: Сотрудник или Администратор
```

## Буферизация медиагруппы

В `userhours.py` три независимых потока буферизации — по одному на каждый FSM-шаг Официанта. Механизм одинаков для всех трёх.

**Механизм:** Telegram присылает фото медиагруппы отдельными апдейтами с одинаковым `media_group_id`. Handler накапливает `file_id` в `_mg_*_photos`, а `asyncio.create_task(_delayed_process_*(mgid))` создаётся **один раз** (guard через `_mg_*_scheduled`). `asyncio.Lock` исключает race condition при параллельных апдейтах. После `asyncio.sleep(1.0)` все фото уже накоплены.

**Очистка** происходит в трёх местах: после успешной обработки, при отмене пользователем и в `except`-блоке (предотвращает утечку памяти).

### Поток 1 — Смена Официанта (`waiting_shift_input`)

| Dict | Назначение |
|------|-----------|
| `_mg_photos[mgid]` | Список `file_id` фото группы |
| `_mg_context[mgid]` | `{message, state, caption}` — контекст handler'а |
| `_mg_scheduled` | Guard-set: предотвращает двойной `asyncio.create_task` |
| `_mg_locks[mgid]` | `asyncio.Lock` для защиты от race condition |

Отложенная функция: `_delayed_process_waiter(mgid)` → `_send_waiter_report()`

### Поток 2 — Карты лояльности (`waiting_loyalty_cards`) *(Stage 16)*

| Dict | Назначение |
|------|-----------|
| `_mg_loyalty_photos[mgid]` | Список `file_id` фото карт лояльности |
| `_mg_loyalty_context[mgid]` | `{message, state}` — контекст handler'а |
| `_mg_loyalty_scheduled` | Guard-set: предотвращает двойной `asyncio.create_task` |
| `_mg_loyalty_locks[mgid]` | `asyncio.Lock` для защиты от race condition |

Отложенная функция: `_delayed_process_loyalty(mgid)`

### Поток 3 — Наполняемость чеков (`waiting_check_filling`) *(Stage 16)*

| Dict | Назначение |
|------|-----------|
| `_mg_filling_photos[mgid]` | Список `file_id` фото чеков |
| `_mg_filling_context[mgid]` | `{message, state}` — контекст handler'а |
| `_mg_filling_scheduled` | Guard-set: предотвращает двойной `asyncio.create_task` |
| `_mg_filling_locks[mgid]` | `asyncio.Lock` для защиты от race condition |

Отложенная функция: `_delayed_process_filling(mgid)`

## Защита от инъекций и атак

### Formula injection (Google Sheets)
- **value_input_option="RAW"** для всех пользовательских данных
- Исключение: вставка hardcoded формул (`=SUMPRODUCT`, `=S+AJ` и др.) использует `USER_ENTERED`
- Защита от: `=HYPERLINK()`, `=IMPORTXML()`, `=IMPORTDATA()`

### HTML injection (Telegram)
- `html.escape()` для всех user inputs в HTML-сообщениях
- Применяется к: `full_name`, комментариям, custom_position
- Функция `make_mention()` в `app/utils/text_utils.py` экранирует имена

### Callback data overflow
- Лимит 64 байта — данные хранятся в dict (`_pending_admins`), не в callback
- Только короткие идентификаторы в callback_data

### Email validation
- Только `@gmail.com` через `_is_valid_gmail()`
- Маскировка в логах через `mask_email()` → `p***r@gmail.com`

### PII в логах
- **НЕ логировать** `full_name` — только `telegram_id`
- Применяется в: `/set_rate`, `/set_rate_all`, approve flow, dismiss flow

## SQLite — дополнительные настройки

### WAL mode + timeout (добавлено при Docker деплое)

```python
async with aiosqlite.connect(DB_PATH, timeout=10) as db:
    await db.execute("PRAGMA journal_mode=WAL")
    ...
```

- **WAL (Write-Ahead Logging)** — несколько читателей + один писатель без блокировок
- **timeout=10** — ждёт до 10 секунд при конкурентном доступе (предотвращает "database is locked")
- Критично при Docker: несколько хендлеров могут обращаться к SQLite одновременно

## Критичные паттерны

### LEFT JOIN для user_rates

При получении списка сотрудников всегда использовать LEFT JOIN:

```sql
SELECT u.*, ur.base_rate, ur.extra_rate
FROM users u
LEFT JOIN user_rates ur ON u.telegram_id = ur.telegram_id
```

**Почему:** INNER JOIN исключает новых сотрудников без ставок → замкнутый круг
(нужна ставка чтобы назначить ставку). Фикс задеплоен в коммите d6a7ce2.

**Реализация:** `get_users_rates_by_department()` в `app/db/models.py` — использует LEFT JOIN.
Для новых сотрудников `base_rate` и `extra_rate` будут `None`.

